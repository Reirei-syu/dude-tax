# 系统维护可编辑税标最小闭环方案

## 1. 背景
- 当前系统维护模块只有只读页面，税标数据仍直接来自 `packages/config` 中的静态常量。
- 首页、计算中心和个税计算核心都依赖同一份静态税标，用户无法在本地修改税标，也无法让修改后的税标参与后续计算。
- 项目当前已具备结果查询、导出和历史查询最小闭环，一旦税标开放编辑，就必须解决“新税标生效后旧结果是否失效”的一致性问题。

## 2. 目标
- 提供“系统维护”模块的最小可编辑闭环，支持查看、编辑、保存当前全局税标。
- 保存后的税标应同时作用于首页展示、系统维护页展示和后续年度计算逻辑。
- 税标变更后，旧年度结果必须失效，避免继续展示旧口径结果。
- 控制改动范围，优先复用现有 `app_preferences` 存储能力，不引入复杂版本体系。

## 3. 设计方案
### 3.1 存储策略
- 在 `app_preferences` 中新增全局税标配置键，保存一份 JSON 结构的税标快照。
- 若数据库中不存在税标配置，则回落到 `packages/config` 默认税标并自动返回默认值。
- 暂不做多版本税标管理，只维护“当前生效税标”一份配置。

### 3.2 Core 层调整
- 在 `packages/core` 中新增税标相关类型：
  - `TaxBracket`
  - `BonusTaxBracket`
  - `TaxPolicySettings`
- 将年度计算核心改为接收税标参数，默认仍可使用静态默认税标，避免调用方一次性全部改造失败。

### 3.3 API 层调整
- 新增税标仓储，负责：
  - 读取当前税标
  - 保存当前税标
  - 返回默认税标
- 新增系统维护路由：
  - `GET /api/tax-policy`
  - `PUT /api/tax-policy`
- `PUT` 保存时执行参数校验：
  - 基本减除费用必须大于等于 0
  - 税率表级数不能为空
  - 税率、速算扣除数、阈值必须满足基本数值约束

### 3.4 结果失效策略
- 采用最小实现：当税标保存成功后，直接清空 `annual_tax_results` 与 `annual_calculation_runs`。
- 保留 `employee_month_records`、员工档案和单位数据不变。
- 这样首页提醒、计算中心和结果中心都会自然进入“需重新计算”的状态。

### 3.5 前端系统维护页
- 在当前只读页基础上升级为“查看 + 编辑”双态页面。
- 首版支持：
  - 编辑基本减除费用
  - 编辑综合所得税率表
  - 编辑年终奖单独计税税率表
  - 重置为默认税标
  - 保存税标
- 首版不做：
  - 富文本说明维护
  - 版本切换
  - 变更审计

### 3.6 首页与其他页面联动
- 首页税率展示改为读取当前税标接口返回值，不再直接绑定静态常量。
- 系统维护页保存成功后，刷新当前页面数据，并提示“年度结果已失效，请重新计算”。
- 计算中心与结果中心不直接改 UI 结构，但依赖被清空后的现有状态自然反映“需重新计算”。

## 4. 涉及模块
- `packages/core`
  - 税标类型
  - 年度计算函数入参
- `packages/config`
  - 默认税标导出结构整理
- `apps/api`
  - 税标仓储
  - 系统维护路由
  - 保存税标后的结果失效处理
- `apps/desktop`
  - 系统维护页编辑表单
  - 首页税率展示改为读取 API

## 5. 数据结构变更
- 不新增业务表。
- 复用 `app_preferences`，新增税标配置键，例如：
  - `tax_policy_settings`
- 不做历史版本表，不做税标版本号表。

## 6. 接口变更
- 新增：
  - `GET /api/tax-policy`
  - `PUT /api/tax-policy`
- 影响：
  - 首页税率显示改为依赖税标读取接口
  - 系统维护页改为真实编辑页

## 7. 风险评估
- 风险 1：税标保存后直接清空结果与重算记录，会使当前历史查询结果立即消失。
  - 说明：这是当前最小方案的代价，用于保证口径一致。
- 风险 2：个税计算核心从静态常量改为可注入税标，涉及 `packages/core` 的核心逻辑入口。
  - 说明：需要用测试锁住默认税标与自定义税标下的行为。
- 风险 3：首页税率表从静态常量改为 API 读取后，会增加首页加载依赖。
  - 说明：需要保留加载态和失败回退提示。

## 8. 回退方案
- 若实施后出现问题，可回退到当前 `master`，恢复只读系统维护页和静态税标模式。
- 若仅保存逻辑有问题，可临时禁用 `PUT /api/tax-policy`，保留 `GET /api/tax-policy` 只读能力。
- 若结果失效策略影响过大，可暂停“保存即清空结果”的实现，回退为只读页，待版本化方案成熟后重做。
