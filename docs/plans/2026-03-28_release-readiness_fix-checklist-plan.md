# Release Readiness 修复清单

## 1. 背景
- 当前项目已经达到“主链可运行、主要模块可联动”的状态，但仍未达到“可试点发布”。
- 全仓审查暴露的问题主要集中在桌面生产运行链路、中文乱码、源码可维护性、结果失效判定、前端越层计算、数据库约束、导入事务与性能、超大页面拆分等方面。
- 本清单目标不是继续扩散需求，而是将现有问题按发布优先级固化为可执行修复序列。

## 2. 目标
- 将项目从“可运行但不可发布”提升到“可试点发布”。
- 严格按 `P0 -> P1 -> P2 -> P3` 顺序推进，不允许跳步。
- 每完成一个子任务，都必须执行：
  - 自行 Review
  - 本地验证
  - 更新 `PROGRESS.md`

## 3. 设计方案

### 3.1 总体策略
- P0：修复桌面生产运行链路，确保 Electron 包、API 与 SQLite 在生产包中真实可运行。
- P1：修复直接影响用户可用性的问题，包括乱码和被压扁的源码。
- P2：修复业务正确性和架构边界问题，包括结果失效判定、前端越层调用、数据库最后防线。
- P3：收口性能与可维护性问题，包括导入事务与查询优化、超大页面拆分、解释逻辑统一。

### 3.2 分阶段范围

#### P0：必须完成才能继续
1. 修复桌面端打包链路（Electron + API 可运行）
2. 移除生产环境对 `tsx + TypeScript` 源码的依赖
3. 修复 `better-sqlite3` 与 Electron ABI 不匹配问题

#### P1：影响用户可用性
4. 全量修复中文乱码（UI + 默认数据 + config）
5. 恢复被压扁的一行源码文件（可维护性）

#### P2：架构和数据正确性
6. 修复历史版本 `isInvalidated` 判定逻辑
7. 将“历史结果重算对比”从前端迁移到 API
8. SQLite 增加 `CHECK` 约束

#### P3：稳定性与可维护性
9. 导入服务改为事务 + 避免 N+1 查询
10. 拆分超大页面
11. 统一文案映射与结果解释逻辑

## 4. 涉及模块
- `apps/desktop`
- `apps/api`
- `packages/core`
- `packages/config`
- `docs/plans`
- `docs/tasks.md`
- `docs/context/latest_context.md`
- `PROGRESS.md`

## 5. 数据结构变更
- P0 不新增业务表，优先解决生产运行链路。
- P2 预计新增：
  - 结果数据签名字段 `data_signature`
  - 若 SQLite 无法直接补 `CHECK`，则采用兼容旧库的迁移方案重建表结构

## 6. 接口变更
- P0：
  - API 产出编译后 `dist/server.js`
  - Electron 改为启动编译后的 JS 入口
- P2：
  - 新增历史结果重算对比接口，如 `/api/history/recalculate`
  - 结果查询结构需补充基于 `policy_signature + data_signature` 的失效判定

## 7. 风险评估
- P0 风险最高：若桌面生产包链路未先稳定，其余问题即使修完也无法试点发布。
- P1 风险中高：乱码和单行源码会持续放大误判和误改风险。
- P2 风险高：涉及结果有效性语义和数据库约束，必须先出方案再实施。
- P3 风险中：不直接阻塞试点，但会影响后续稳定迭代。

## 8. 回退方案
- 每个阶段只做一个明确收口目标，避免跨阶段混做。
- P0 和 P2 涉及架构 / 数据结构调整时，必须先形成对应方案文档再执行。
- 若某阶段失败：
  - 回退该阶段变更
  - 更新 `docs/tasks.md`
  - 在 `PROGRESS.md` 记录阻塞原因
