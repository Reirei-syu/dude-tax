# 剩余功能模块收口计划

## 1. 背景
- 当前项目主模块已基本跑通，但仍有若干“已规划 / 已半落地 / 已具备方案文档”的功能没有真正闭环。
- 用户要求优先将未开发的功能模块及其子功能模块继续推进完成。
- 结合 `PENDING_GOALS.md`、`docs/tasks.md` 与当前代码状态，剩余重点不在新增大模块，而在补齐现有半闭环能力和复杂业务子模块。

## 2. 目标
- 将当前未闭环的功能按优先级拆成可持续执行的小任务。
- 优先打通已经进入 `core` 但尚未进入 DB / API / UI 的补发补扣链路。
- 继续推进复杂预扣预缴规则、跨单位/跨年衔接和批量导出增强。

## 3. 设计方案
- 采用“三阶段收口”：
  - 阶段 A：补发补扣字段全链路闭环
  - 阶段 B：复杂个税口径继续落地
  - 阶段 C：批量导出体系增强
- 执行顺序同时遵守项目优先级：
  - `core -> service -> ui`
  - 先打通数据结构和接口，再开放页面入口

## 4. 涉及模块
- `packages/core`
- `apps/api`
- `apps/desktop`
- `docs/tasks.md`
- `PROGRESS.md`
- `docs/context/latest_context.md`

## 5. 数据结构变更
- 为月度记录增加补发补扣相关持久化字段：
  - `supplementary_salary_income`
  - `supplementary_withheld_tax_adjustment`
  - `supplementary_source_period_label`
  - `supplementary_remark`
- 导入模板与接口结构需同步补齐。

## 6. 接口变更
- 月度录入接口补齐补发补扣字段收发。
- 必要时同步补齐快速计算和批量导入的输入结构。

## 7. 风险评估
- 补发补扣字段一旦进入持久化链路，会影响结果失效、重算和导入模板。
- 完整预扣预缴规则属于高风险税务逻辑，必须保持小步验证。
- 批量导出多员工 / 多年度会拉高导出性能和文件体积风险。

## 8. 回退方案
- 每项任务保持原子化，优先以字段补齐、接口补齐、UI 开关式接入为主。
- 如某一阶段影响主链路，可单独回退对应任务文件，不影响其他模块。

## 9. 任务拆解
- [ ] 月度录入：补发补扣字段持久化与 API 打通
- [ ] 月度录入：补发补扣导入模板与导入链路补齐
- [ ] 月度录入：补发补扣 UI 区块与校验
- [ ] 个税计算核心：完整预扣预缴规则落地
- [ ] 个税计算核心：跨单位 / 跨年衔接规则
- [ ] 批量导出：多员工 / 多年度导出能力
