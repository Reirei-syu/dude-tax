# 录入模型精简优化方案

## 1. 背景

当前年度工作台仍遗留“补发收入 / 补扣税调整 / 记录状态”等旧模型，和新的“确认月份”主流程存在重复语义与理解成本。

## 2. 目标

- 统一“其他收入”语义
- 取消“记录状态”对计算的控制
- 让已确认月份天然成为有效月份
- 在工作台中展示只读减除费用列

## 3. 设计方案

- core 收缩公开月度 DTO，工作台与计算都使用 `otherIncome / otherIncomeRemark`
- API 保留旧字段入参兼容，但新主流程只走新字段
- 结果确认允许空白月确认，并在确认结果与历史查询中按零值月参与计算
- 前端工作台删除状态列，新增只读减除费用列

## 4. 涉及模块

- `packages/core`
- `apps/api`
- `apps/desktop`
- `docs`

## 5. 数据结构变更

- 不新增表
- 保留旧数据库列做兼容
- 新主流程不再依赖：
  - `status`
  - `supplementary_withheld_tax_adjustment`
  - `supplementary_source_period_label`

## 6. 接口变更

- 月度工作台保存接口改为无状态字段
- 快速计算 payload 改为无状态字段
- 年度工作台、结果确认、历史查询明细改为 `otherIncome / otherIncomeRemark`

## 7. 风险评估

- 中风险：旧导入模板、旧单月接口和旧测试仍依赖 legacy 字段
- 通过“输入兼容 / 主流程切新口径 / legacy 路由回读别名”方式平滑过渡

## 8. 回退方案

- 不删库列
- 不删旧单月接口
- 一旦新字段链路出问题，可回退到本次提交前版本继续使用旧兼容字段
