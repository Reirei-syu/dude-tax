# 跨单位 / 跨年衔接规则方案

## 1. 背景
- 当前个税计算核心以“当前单位 + 当前年度”的月度记录为单一计算输入。
- 这能支撑最小闭环，但在真实工资薪金场景下仍缺少两类关键衔接：
  - 同一纳税人年度内跨单位任职时，累计预扣不应被错误重置
  - 上一年收入、发薪起始月等规则上下文不能只看当前单位局部数据
- 当前仓储层已具备员工 `idNumber`、入职/离职日期、单位/年度维度月度记录等基础数据，可支撑衔接建模。

## 2. 目标
- 明确跨单位 / 跨年衔接的计算边界与首版实现策略。
- 在不推翻现有 `单位 + 年份` 工作房间模型的前提下，为同一身份证号建立规则连续性。
- 将任务拆成可验证的小步，不在一次迭代中同时改完所有税务口径。

## 3. 设计方案

### 3.1 核心原则
- 正式计算结果仍以“当前单位 + 当前年度”为展示房间。
- 但预扣规则上下文允许引用：
  - 同一 `idNumber` 在同一自然年度内其他单位的已完成月度记录
  - 同一 `idNumber` 在上一完整纳税年度的全年收入概览
- 展示和导出要明确区分：
  - 本单位结果
  - 规则衔接时引用的外部上下文

### 3.2 首版实现范围
- 同年度跨单位衔接
  - 对同一 `idNumber`，在计算当前单位某年度结果时，允许把同年度其他单位已完成月份作为“累计预扣前置月份”。
  - 这些月份只参与累计预扣轨迹和规则判断，不直接并入当前单位展示的工资收入合计。
- 上一年度收入衔接
  - 计算 `previousYearIncomeUnder60k` 时，不再只依赖手工上下文，而是优先从同一 `idNumber` 上一完整纳税年度的全部单位汇总判断。
- 首次取得工资月份衔接
  - 对同一 `idNumber` 在当前年度的所有单位月度记录，取最早的已完成发薪月份，作为 `firstSalaryMonthInYear` 的默认推导值。

### 3.3 不纳入首版
- 自动处理往期更正申报
- 跨年度补发所属期反算
- 多身份合并（身份证号缺失或变更）
- 单位间重复月度记录自动冲突裁决

### 3.4 数据与仓储调整建议
- 新增仓储查询能力：
  - 按 `idNumber + taxYear` 聚合同一年度跨单位月度记录
  - 按 `idNumber + previousTaxYear` 汇总上一年度收入
- 不新增表，优先复用：
  - `employees.id_number`
  - `employee_month_records`
- 若后续需要性能优化，再考虑补充年度级聚合缓存表。

### 3.5 核心算法调整建议
- 在核心层新增“规则衔接上下文构建器”，先把跨单位 / 跨年外部数据整理为：
  - `carryInCompletedRecords`
  - `derivedPreviousYearIncomeUnder60k`
  - `derivedFirstSalaryMonthInYear`
- `calculateEmployeeAnnualTax` 本身仍只负责计算，不直接读库。
- 服务层负责：
  - 读取当前单位员工月度记录
  - 读取同一 `idNumber` 的跨单位 / 上年上下文
  - 构造衔接上下文后再调用核心算法

### 3.6 前端影响
- 计算中心
  - 重算时需提示“当前结果已引用跨单位 / 上年规则上下文”
- 结果中心
  - 需补充“规则衔接说明”，明确是否引用了其他单位或上一年度数据
- 历史查询
  - 差异对比后续要增加“规则来源变化”维度

## 5. 数据结构变更
- 首版不新增表
- 仅新增仓储查询方法与核心上下文类型

## 6. 接口变更
- 年度重算结果返回结构可增加“规则衔接摘要”
- 首版建议不新增独立 API，先复用现有结果结构

## 7. 风险评估
- 风险等级：高
- 主要风险：
  - 同一身份证号跨单位月度记录若存在重复月份，累计预扣会被重复计入
  - 当前“单位房间”心智与“规则引用其他单位数据”之间存在认知冲突，需要强提示
  - 若一次性并入上一年度收入判断、首次发薪判断和同年跨单位累计，测试矩阵会迅速膨胀

## 8. 回退方案
- 首版保持“服务层构造上下文、核心层纯计算”的结构，回退时只需移除服务层衔接构造逻辑
- 因不新增表，回退不涉及数据库回滚
