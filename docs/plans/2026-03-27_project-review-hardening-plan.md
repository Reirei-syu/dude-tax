# 全项目审查与加固方案

## 1. 背景
- 当前项目已具备桌面端最小闭环，但本轮用户要求对全项目做一次面向生产可用性的系统审查。
- 审查目标不仅包括代码漏洞，也包括架构边界、数据交互链路、长期维护性、稳定性与可回退性。
- 结合现状扫描，已发现若干高优先级风险：本地 API 暴露面过宽、导入链路 CSV 解析不稳健、部分源码/提示语存在编码异常、文档中安全状态与实际依赖审计结果不一致。

## 2. 目标
- 完成一次覆盖前端、桌面壳、本地 API、核心计算与数据库交互的结构化审查。
- 修复可高确定性落地的安全与稳定性问题。
- 输出可执行的剩余风险清单，避免“表面通过测试、实际存在长期隐患”。

## 3. 设计方案
- 审查维度拆为四类：
  - 安全暴露面：CORS、Electron 窗口权限、IPC 边界、依赖漏洞。
  - 数据交互：UI -> API -> Repository -> DB -> Core 的参数一致性、结果一致性、异常链路。
  - 稳定性：导入解析、边界输入、错误提示、数据持久化和回退能力。
  - 可维护性：分层边界、包引用方式、未完成能力与已上线能力的一致性。
- 修复策略采用“高确定性优先”：
  - 直接修复：本地 API CORS 白名单、Electron 窗口安全选项、导入 CSV 解析器稳健性、明显乱码提示。
  - 记录不立即大改：跨包深相对路径导入、补发补扣字段全链路未闭环、依赖树中的上游漏洞。

## 4. 涉及模块
- `apps/api`
- `apps/desktop`
- `packages/core`
- `docs/plans`
- `docs/tasks.md`
- `PROGRESS.md`
- `docs/context/latest_context.md`

## 5. 数据结构变更
- 本轮优先不引入新的业务表结构。
- 如需支持审查修复，仅允许做非破坏性的配置与解析逻辑调整。

## 6. 接口变更
- 允许收紧本地 API 的 CORS 策略，仅放行桌面端与本地开发所需来源。
- 不变更现有业务接口路径与返回结构。

## 7. 风险评估
- CORS 收紧后，若遗漏合法来源，可能影响本地调试或 Electron 渲染层访问。
- CSV 解析器增强后，若兼容性处理不完整，可能影响既有导入模板行为。
- Electron 安全选项显式化后，若与现有 preload 行为不兼容，可能影响导出等桌面能力。

## 8. 回退方案
- CORS、Electron、导入解析均保持单点文件修改，可逐文件回退。
- 所有修复通过测试与类型检查验证；若某项加固影响主链路，可仅回退对应提交范围。

## 9. 任务拆解
- [ ] 审查修复：本地 API CORS 白名单与 Electron 窗口安全加固
- [ ] 审查修复：导入链路 CSV 解析稳健性增强
- [ ] 审查修复：乱码提示收口与审查文档同步
- [ ] 审查验证：运行测试、类型检查与依赖审计，输出剩余风险
