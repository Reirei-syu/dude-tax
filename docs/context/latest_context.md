## 关键设计决策
- 补发补扣字段当前只进入共享类型和 helper，不直接并入现有算法主链
- 补发补扣首版仍按“支付当月补发 / 调整”建模，不先做往期申报更正工作流
- 继续保持 `UI -> Service/API -> Core` 分层，不在前端直接拼后端业务规则
- 本地 API 仅允许 Electron / 本地开发来源访问，外部站点一律返回 `403`
- Electron 窗口禁止打开新窗口和导航到外部页面，外链统一交给系统浏览器
- 补发补扣先按“支付当月补发 / 补扣”建模，先做当前月输入闭环，再继续扩展更复杂税务场景
- 快速计算模板化输入采用“当前月份即模板”的纯前端批量套用方案，不新增后端接口
- 预扣规则上下文先从快速计算页接入，作为完整预扣预缴情形落地的第一条使用链路
- 正式年度重算已允许显式传入预扣规则模式，并将模式写入结果快照
- 预扣规则说明统一由前端 helper 生成，结果中心和历史查询复用同一套解释逻辑
- 跨单位 / 跨年衔接首版采用“服务层构造上下文、核心层纯计算”的策略，不直接让核心读库
- 跨单位衔接上下文当前已可推导：相关员工集合、上一年度收入是否不超过 6 万元、当前年度首次发薪月、同年度其他单位已完成记录
- 核心算法当前只把“前置月份”并入累计预扣，不把外部单位数据直接并入当前单位展示摘要
