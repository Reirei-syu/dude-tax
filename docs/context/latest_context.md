# 当前上下文摘要

## 当前阶段
- 项目处于 `v0.1.0-alpha`，主业务闭环已覆盖单位管理、员工信息、月度录入、年度计算、结果查看、历史查询最小闭环，以及 CSV / XLSX 双格式导出。
- 当前代码已通过 `npm run typecheck` 与 `npm test` 验证，本轮已将税标变更后的结果处理从“物理删除”升级为基于 `policy_signature` 的逻辑失效。

- Electron + React + Fastify + SQLite 本地桌面骨架可运行。
- “单位 + 年份” 全局上下文闭环已接通。
- 员工信息、月度录入、年度计算、结果中心主链路已跑通。
- 结果中心支持手动切换计税方案、导出字段选择、导出模板、CSV / XLSX 双格式导出。

- 历史查询已支持切换查看当前有效结果、已失效快照和全部结果。
- 历史查询已支持导出当前筛选结果的 CSV / XLSX。
- 历史查询已支持对已失效快照按当前税标进行结果差异对比。
- 历史查询已支持更完整的摘要统计，包括应退税、已平、单位数、员工数和年份数。
- 历史查询已支持跨年度概览与按年度快速切换。
- 年度结果导出与历史查询导出都已支持基础表头样式、列宽和冻结表头。
- 年度结果导出与历史查询导出都已支持多工作表（结果页 + 导出说明页）。
- 桌面端导出已支持原生“另存为”文件落地，浏览器环境会自动回退为下载。
- 年度结果导出与历史查询导出都已支持基础 Excel 样式模板。
- 年度结果导出与历史查询导出都已支持分批写入与异步 Buffer 构建，以降低大结果集导出阻塞。
- 月度数据录入已支持复制上月内容到当前月份。

- 系统维护已支持查看、编辑和保存当前税标与全局说明，并在税标变更后逻辑失效旧结果。
- 首页工作提醒已联动“待重算”和“未完成月份”统计。
- 首页已根据上下文、员工准备状态和重算状态给出工作建议入口。
- 首页税率表已改为读取当前生效税标，而不是直接读取静态默认配置。
- 保存税标后，旧结果不再被物理删除，而是按 `policy_signature` 逻辑判定为失效。
- `PENDING_GOALS.md` 已沉淀未完成目标清单，便于后续迭代拆解。

## 关键设计决策
- 继续遵守最小改动原则，不调整既有分层边界，仍保持 `UI -> Service/API -> Core` 的调用方向。
- 历史查询阶段先读取最新年度结果快照，不引入历史版本重算链路，避免本轮扩大范围。
- 税标当前通过 `app_preferences.tax_policy_settings` 保存单一生效版本，不引入复杂版本表。
- 全局说明当前通过 `app_preferences.tax_policy_maintenance_notes` 保存单一文本，不做版本化和富文本结构。
- 税标保存后采用基于 `policy_signature` 的逻辑失效策略，优先保留旧结果数据并只过滤当前有效结果。
- 仓库整理采用“临时分支提交 -> 合并回 master”的方式，而不是直接在脏工作区提交，优先保证可回退性。

## 未解决问题
- 当前仓库没有配置远程仓库，后续若要协作或备份，需要补充 `origin` 与分支协作规范。
- 系统维护仍不支持税标版本管理、富文本说明维护和更细粒度结果失效联动。
- 首页工作提醒中的“导入冲突待处理”仍是占位值，需等批量导入模块落地后再联动。
- 首页工作建议当前仍基于规则推荐，未建立更细粒度的优先级模型。
- 历史查询仍不包含重算历史版本能力。
- 月度录入仍缺少批量编辑、批量保存和数据差异提示。
- 结果中心仍缺少历史版本查看、版本差异对比和导出模板管理。
- `exceljs` 引入后桌面端 bundle 体积偏大，后续要评估按需加载或代码拆分。
